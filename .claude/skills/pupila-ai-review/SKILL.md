`pnpm run ai-review` is a **local-only** companion that augments selected jobs with an LLM review via `src/lib/llm.ts` (auto-detects `claude` / `codex` / `gemini` / `opencode`, override `PUPILA_LLM`). Uses the user's local subscription (e.g. Claude Max) ŌĆö NOT an API key, so no per-token charges.

The launchd/cron review agent runs daily at 07:15 by default. Without an LLM CLI, run `scripts/install-launchd.sh --no-review` (or cron equivalent).

## Inputs / outputs

| Input | Path | Role |
|---|---|---|
| Slim job list | `data/jobs.json` | Job metadata, scoring, _signals |
| Body sidecar | `data/jobs-bodies.json` | Full body text (regenerated by `pnpm run dev`) |
| Existing reviews | `data/ai-reviews.json` | Prior runs (incremental ŌĆö only review missing IDs) |
| Candidate brief | `config/candidate-brief.md` | **The main tuning lever** ŌĆö natural-language candidate description embedded verbatim in the prompt |

| Output | Path | Schema |
|---|---|---|
| Reviews | `data/ai-reviews.json` | `Record<jobId, AiReview>` |

Each `AiReview` carries:

- One-sentence `summary`
- 3 bullets each for `wants` / `offers` / `redFlags`
- `verdict` Ōłł `strong-match | match | weak-match | skip`
- One-sentence `reason`

**Writes after every successful review** ŌĆö Ctrl-C or rate-limit kill leaves a partial-but-valid file.

## Selection logic

Default: top 20 by `fitScore` not already reviewed. Reviews for jobs no longer in `jobs.json` are pruned each run.

Flags:

- `--top=N` ŌĆö change the batch size
- `--force` ŌĆö re-review even if an entry exists
- `--ids=a,b,c` ŌĆö explicit list (overrides `--top`)

## JSON parsing (the fragile part)

The LLM occasionally wraps JSON in markdown fences (```` ```json ... ``` ````). `src/ai-review-parse.ts`:

1. Strips fences (recognises ` ```json ` and bare ` ``` `).
2. Falls back to safe defaults on parse failure rather than throwing ŌĆö **partial reviews are still useful**, don't crash the whole run.
3. Sanitizes dirty arrays (empty strings, non-string entries).
4. Validates verdict against the 4-value enum; invalid ŌåÆ coerces to `weak-match`.

Tests in `tests/ai-review-parse.test.ts` (9 cases) cover all the failure modes seen in practice. **When you add a new format quirk, add a fixture.**

## The candidate brief

`config/candidate-brief.md` is the **only natural-language config in the repo**. Hand-edited, gitignored. Generated via `pnpm run setup-brief --file ~/cv.pdf` (or via the UI's Profile tab ŌåÆ drop a PDF/DOCX/MD CV). The CLI shells out to whichever local LLM CLI is installed (`claude` / `codex` / `gemini` / `opencode` ŌĆö auto-detected; override via `PUPILA_LLM=<provider>`).

The brief is embedded **verbatim** in the review prompt ŌĆö it's the main lever for tuning match/skip behaviour. To change verdicts at scale, edit the brief; to change individual scores, see the `pupila-filters` skill.

Template: `config/candidate-brief.example.md` (committed).

## AI Apply (per-job tailored package)

UI rows have an `AI Apply Ō£©` button next to `Apply ŌåŚ`. POSTs `/api/ai-apply` with `{ jobId }` ŌĆö runs the LLM CLI on (brief + posting + CV) and writes a tailored package to `data/applications/<jobId>.md`. Auto-marks as applied.

**Architecture:**

- Middleware: `aiApplyApiPlugin` in `ui/vite.config.ts`.
- Core: `runAiApplyForJob` in `src/lib/ai-apply.ts` ŌĆö same function used by:
  - The Vite middleware (single job, synchronous via spawn)
  - The apply-worker (`scripts/apply-worker.ts`, drains the Jinder swipe queue serially)
- Validation: `isValidJobId` from `src/lib/apply-queue.ts` (sha1 hex) gates every entry point ŌĆö `data/applications/<jobId>.md` writes are path-traversal targets otherwise.

## Debugging "verdicts feel wrong"

1. **Re-read the brief.** Most miscalibration traces to the brief, not the prompt. If you're saying "skip" for things you'd actually take, the brief over-specifies what to avoid.
2. **Check the input.** `jq '.[\"<jobId>\"]' data/ai-reviews.json` ŌĆö what `wants` / `offers` / `redFlags` did the model extract? Often the body was truncated or boilerplate-heavy.
3. **Check the model.** Different providers produce different verdict distributions; `claude` and `codex` are the most stable in our experience.
4. **Force-rerun a single job:** `pnpm run ai-review --force --ids=<jobId>` ŌĆö read the new output, iterate on brief.

## Wiring a new LLM provider

`src/lib/llm.ts` exports `detectLlmCli()` + `runLlm()`. To add a new provider:

1. Add detection logic (PATH lookup) in `detectLlmCli()`.
2. Add command-line invocation in `runLlm()` (must support stdin prompt + stdout response).
3. Add to the provider enum in `ui/src/types.ts` if it appears in the LLM-picker UI.
4. Test via Settings ŌåÆ LLM CLI ŌåÆ "Test connection" (6-token prompt, 30s timeout).
