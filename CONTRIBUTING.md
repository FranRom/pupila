# Contributing

This repo is a local-first, config-driven job aggregator. Contributions should preserve that shape: fetch public sources, normalize jobs, score from local config, write local files, and avoid adding hosted services or databases.

## Before You Start

- Read [`README.md`](./README.md) for the product flow and [`AGENTS.md`](./AGENTS.md) for the detailed implementation rules.
- Use Node 22 and pnpm 10.
- Keep changes small and focused. Do not mix personal profile tuning, generated job output, and code changes unless the PR explicitly needs all of them.
- The repo has no open-source license. These notes are a contribution workflow, not a redistribution grant.

## Local Setup

```bash
pnpm install
pnpm run dev
pnpm run ui
pnpm test
```

Use `pnpm run daily` only when you intentionally want to run both aggregation and the optional AI review. The AI review shells out to a local CLI (`claude`, `codex`, `gemini`, or `opencode`) through `src/lib/llm.ts`; it must stay local-only.

## Project Invariants

- Local-first only: do not add a hosted scheduler, public dashboard deployment, database, queue, external backend, or tracking service without an explicit project decision.
- `README.md` is hand-maintained. Do not make the pipeline rewrite it.
- `JOBS.md`, `data/jobs.json`, and `data/feed.xml` are generated by `pnpm run dev`; avoid hand-editing them.
- `data/raw/*` and `data/jobs-bodies.json` are transient debugging/AI-review artifacts and stay gitignored.
- The UI is intentionally local-only on `127.0.0.1:5173`. Do not add a deploy command or GitHub Pages workflow unless explicitly requested.
- Applied jobs remain visible in normal job sections. Do not filter them out of the main list.
- `config/applied.json` and `config/candidate-brief.md` can contain personal data. Treat changes to them as intentional user data changes.

## Configuration Changes

Most personalization should be a config edit, not code:

- Add or remove tracked companies in [`config/slugs.json`](./config/slugs.json).
- Tune scoring weights and keyword lists in [`config/profile.json`](./config/profile.json) — or let `/api/profile-generate` regenerate it from `config/candidate-brief.md`.
- Personal data (brief, applied history, raw CV) lives in gitignored `config/` files only. Don't commit example/personal config to the repo.

If a filter regex, hard-drop rule, or scoring behavior changes, update the matching tests in `tests/` in the same change.

## Adding A Source

1. Add the raw source shape and source literal in [`src/types.ts`](./src/types.ts).
2. Create `src/fetchers/<name>.ts` returning `Promise<FetcherResult<Raw>>`.
3. Catch fetcher errors internally and return `{ items: [], errors: [...] }`; fetchers should not throw.
4. Use `fetchWithTimeout`, `fetchJson`, `fetchText`, and the shared request headers from [`src/utils.ts`](./src/utils.ts).
5. For multi-slug fetchers, use `fetchMultiSlug` from [`src/fetchers/_shared.ts`](./src/fetchers/_shared.ts).
6. Add `normalize<Name>` in [`src/normalize.ts`](./src/normalize.ts).
7. Wire it into [`src/index.ts`](./src/index.ts), [`src/dedup.ts`](./src/dedup.ts), and [`src/render.ts`](./src/render.ts).
8. Add focused parser/normalizer tests, especially for HTML or GraphQL scrapers.

Smoke-test before wiring the source into the orchestrator:

```bash
npx tsx -e "import('./src/fetchers/<name>.ts').then(async m => { const r = await m.fetch<Name>(); console.log('count:', r.items.length, 'errors:', r.errors, 'first:', r.items[0]); })"
```

## Code Standards

- TypeScript stays strict and ESM/NodeNext.
- Prefer native `fetch` and existing helpers. Avoid new runtime dependencies unless the tradeoff is clear.
- Keep parsers deterministic and testable; split pure parsing logic from CLI/orchestrator side effects when practical.
- Use structured parsing APIs when available. Avoid brittle string manipulation for structured data.
- Preserve URL safety checks. Upstream URLs must be `http` or `https`.
- Keep generated markdown links escaped and external links protected with `rel="noopener noreferrer"` where raw HTML is used.
- UI changes should follow the existing Vite/React single-app shape and keep the dashboard dense, local, and work-focused.

## Checks

Run the relevant checks before opening a PR:

```bash
pnpm run lint
pnpm run typecheck
pnpm test
pnpm run build
pnpm run ui:build
```

The pre-commit hook runs `pnpm run lint && pnpm run typecheck`. CI also runs lint, typecheck, tests, the TypeScript build, the Vite UI build, and a production dependency audit.

## GitHub Actions

Keep `.github/workflows/check.yml` as a PR/push validation workflow only. Daily aggregation runs locally through `scripts/install-launchd.sh` or `scripts/install-cron.sh`.

If you update GitHub Actions:

- Pin third-party actions by full commit SHA.
- Keep `permissions: contents: read` unless a write permission is explicitly justified.
- Do not restore hosted daily aggregation unless the local-first scheduling model is deliberately changed.

## Commit Style

Use these prefixes:

| Prefix | Use for |
|---|---|
| `feat:` | new fetcher, filter rule, UI behavior, or output section |
| `fix(fetchers):` | upstream URL change or parser bug |
| `fix(filters):` | scoring or hard-drop behavior |
| `test:` | test-only changes |
| `chore:` | scaffolding, dependency bumps, generated snapshots |
| `ci:` | workflow changes |
| `docs:` | README, AGENTS, or CONTRIBUTING changes |

In PR notes, include what changed, how you tested it, and whether generated files were refreshed.
